本仓库是基于 Minima 的 Jekyll/GitHub Pages 博客，整体为纯静态站点，不存在服务端中间件、异常类型体系或 panic/recover 机制。错误处理仅体现在以下三个层面：

1. HTTP 错误页面
- `404.html`：使用 Jekyll 布局渲染“未找到”页面，标题文案为“404 啦，还想干啥”，属于 GitHub Pages 默认行为。
- `ie.html`：针对不支持的现代浏览器特性提供降级提示页，引导用户切换到 Chrome/Firefox 或通过二维码在移动端访问。

2. 前端运行时错误（搜索功能）
- `assets/js/search.js` 负责全文搜索索引的预加载与查询。对 `fetch(searchIndexUrl)` 的失败路径统一通过 `.catch()` 捕获，并将结果区替换为带 `.search-error` 类的提示文本“无法加载搜索索引”。该样式在 `assets/css/search.css` 中定义，用于以灰色居中文字呈现降级信息。
- 多处重复了相同的 catch 逻辑（索引预加载、首次请求、后续请求），形成一致的前端降级策略：网络或构建产物缺失时不抛错，而是静默回退到空结果并提示用户。

3. 插件与构建期钩子
- `_plugins/*.rb` 中的 Ruby 脚本均为 Jekyll 构建期辅助工具（检查图片、校验年份目录、转义 Liquid、规范化代码语言等），未发现自定义异常类或显式 `raise` 的错误传播模式；若构建期出现异常，通常由 Jekyll 自身抛出堆栈。

结论：该仓库没有统一的错误类型、错误码或中间件层，错误处理遵循“最小化 + 友好降级”的静态站点惯例——HTTP 级用 404/ie 页面，前端异步操作用 try/catch 包裹并以 UI 提示替代崩溃。
